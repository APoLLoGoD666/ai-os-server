## Memory Inventory

### 1. Episodes (`agent-system/episodic-memory.js`)
**What:** Structured JSON records of every pipeline run. Task, outcome, stage, cost, duration, failed stage, models used.  
**Where:** Vault `12 Memory/Episodes/ep-{id}.json`  
**Cap:** 200 files (oldest pruned when exceeded)  
**Retrieval:** Keyword overlap × 0.7 + recency × 0.3  
**Access:** In-process cache (50 entries), full disk load on cache miss  
**Maturity:** Strong — operational since v1 certification

**Gaps:**
- No semantic retrieval (pgvector not wired)
- No `lesson` field on episodes (reflector generates lesson but doesn't write back to episode)
- No per-complexity-tier grouping for pattern analysis
- 200-episode hard cap may be too low for long-term learning (~6 months at 1 run/day)

---

### 2. Lessons (`agent-system/obsidian-memory.js` → `01 Executive/Lessons.md`)
**What:** One-sentence actionable engineering lessons generated by Haiku after each pipeline run.  
**Where:** Obsidian vault Markdown file + Supabase `apex_lessons` table  
**Retrieval:** `reflection-engine.js::getRankedLessons()` — keyword + recency ranking, top 8 injected  
**Consolidation:** `reflection-engine.js::consolidateLessons()` available but no cron wired yet  
**Maturity:** Moderate — generation works, consolidation not automated

**Gaps:**
- Lessons.md grows unboundedly without the consolidation cron
- No lesson deduplication (identical lessons accumulate)
- No lesson ID/versioning (positional only — loses history when consolidated)
- Lesson quality varies: no validation that generated lesson meets quality bar before write
- No usage tracking: no signal on which lessons were semantically relevant to tasks

---

### 3. Reflections (`agent-system/reflection-engine.js`)
**What:** Analytical summaries generated by `generateReflectionLesson()` — deeper synthesis than one-line lessons.  
**Where:** Not currently written to disk — returns a string, caller discards it.  
**Maturity:** Weak — analysis capability exists but output is not persisted anywhere.

**Gaps:**
- `generateReflectionLesson()` is never called from REFLECTOR (wired as comment in cognition-certification.md)
- No reflection storage schema (no vault path, no Supabase table)
- Reflections could replace lessons as the primary learning unit (richer, more actionable)

---

### 4. Stage Performance (`agent-system/agent-reputation.js`)
**What:** Per-stage success rates, latencies, failure rates, error signatures.  
**Where:** Supabase `apex_agent_stages` (persistent) + in-memory cache (5-min TTL)  
**Maturity:** Strong — operational, used by shouldPreEscalate()

**Gaps:**
- No temporal dimension: can't detect "DEVELOPER failure rate increased this week"
- Only 300-row limit queried — long-term trend analysis impossible
- No correlation with task domain (can't distinguish git failures from schema failures)

---

### 5. Conversation Memory (`agent-system/langchain-memory.js`)
**What:** Last 20 turns verbatim + rolling Haiku-generated summary of older turns.  
**Where:** Supabase `apex_lc_sessions`  
**Maturity:** Strong — not a target for this strategy (separate from agent cognition memory)

## Recommended Consolidation Policies

### Policy 1: Lesson Consolidation (Weekly, Sunday 03:30)
**Trigger:** Weekly cron, after adaptation cycle  
**Logic:**
```
1. Load full Lessons.md
2. IF len(lessons) <= 25: skip (no consolidation needed)
3. reflection-engine.js::consolidateLessons(raw, maxOutput=30)
   - Always keep newest 12 (recency guarantee)
   - Score remaining by composite (confidence × 0.4 + recency × 0.3 + actionability × 0.3)
   - Keep top 18 by score
4. Write consolidated 30 lessons back to Lessons.md
5. Archive full pre-consolidation content to vault 12 Memory/Lesson-Archives/lessons-{date}.md
6. Log to reports/consolidation-log-{date}.md
```
**Risk:** Low. Archive before overwrite. Fully reversible.

### Policy 2: Episode Retention (Capacity-Based + Age-Based)
**Current:** 200 episodes, pruned by file mtime (oldest first).  
**Recommended:**
```
Hard cap:           500 (increase from 200 — 1.5 years of coverage at 1 run/day)
Pruning strategy:   Tiered — NOT just "oldest first"
  Tier 1: KEEP always — any episode where success=false AND failedStage != null (failure intelligence)
  Tier 2: KEEP always — any episode in top 10% relevance score for active task domains
  Tier 3: PRUNE first — success=true AND recency > 180 days AND no lessons generated
Pruning trigger:    When count > 500 (not on every write — only when cap exceeded)
```
**Why:** Successful old episodes have low marginal value. Failed episodes are high value forever — they encode system limitations.

### Policy 3: Stage Stats Retention (Time-Based)
**Current:** `apex_agent_stages` queried with `.limit(300)` — no retention policy.  
**Recommended:**
```
Retain:             Last 1000 rows per stage (enough for trend analysis)
Archive:            Weekly snapshot of stage stats to apex_stage_stats_weekly table
Prune:              apex_agent_stages older than 90 days (via server.js cron)
```
**Why:** 90-day window sufficient for adaptation engine. Long-term trend is captured in weekly snapshots.

### Policy 4: Reflection Persistence (NEW — not yet implemented)
**Recommended storage:** `12 Memory/Reflections/` in vault, one file per reflection  
**Schema:**
```json
{
  "id": "ref-{timestamp}",
  "episodeId": "ep-{id}",
  "timestamp": "2026-06-06T03:00:00Z",
  "taskObjective": "...",
  "outcome": "success|failure",
  "synthesizedLesson": "...",
  "failurePattern": "...",
  "successPattern": "...",
  "confidence": 0.85,
  "model": "claude-haiku-4-5-20251001"
}
```
**Retention:** Keep all reflections indefinitely (small files, high signal). Cap at 1000 total.  
**Consolidation:** Monthly — merge reflections into meta-reflections (cross-run synthesis).

## Pruning Strategies

### What to Prune Aggressively
| Memory | Condition | Action |
|--------|-----------|--------|
| Lessons.md | > 30 lessons | Consolidate → archive |
| Old success episodes | > 180 days AND score < 0.4 | Delete |
| apex_agent_stages | > 90 days old | Delete (data in weekly snapshots) |
| cognition-weights.json overrides | Expired OR confidence < 0.5 | Remove override |

### What to NEVER Prune
| Memory | Reason |
|--------|--------|
| Failure episodes with failedStage != null | Permanent failure intelligence |
| Lessons with actionScore = 1.0 (both flags) | Highest-value guidance |
| Weekly stage stats snapshots | Historical trend baseline |
| Reflections | Too small, too valuable |

## Memory Cohesion — Linking Across Types

Currently, the four memory types are **unlinked**. An episode has no reference to the lesson it generated. A lesson has no reference to which stage failed.

### Recommended: Cross-Reference IDs

**Episode → Lesson:**
Add `lessonId` field to episode JSON. After `_reflector()` writes the lesson to Lessons.md, call `episodic.updateEpisode(id, { lessonText, lessonId })`.

**Episode → Reflection:**
Add `reflectionId` field to episode JSON. After generateReflectionLesson() writes to vault, update episode.

**Lesson → Stage:**
Add `generatedByStage` field to lesson (currently not present). Allows: "show me all lessons generated from COMMITTER failures."

**Stage Stat → Episode:**
`apex_agent_stages` rows should include `episode_id` (foreign key to a future `apex_episodes` Supabase table). Enables: "for this episode, which stages failed and when?"
