**Author:** APEX Memory Architecture Engineer  
**Date:** 2026-06-06

## Problem Statement

Memory was **stored** but not **useful**. Specifically:

| Issue | Root Cause | User Impact |
|-------|-----------|-------------|
| Episodes retrieved by keyword | `getSimilarExperiences()` used set intersection on `_keywords(text)` — only exact word matches | "Add pagination" task couldn't find "Add route limiting" episode despite semantic similarity |
| Lessons retrieved by recency | `getRecentLessons(12)` returned last 12 chronologically — no relevance | Auth-related lessons shown for unrelated git tasks, wasting token budget |
| No cross-memory search | Episode retrieval and lesson retrieval were separate calls with no unified interface | ARCHITECT couldn't ask "what's everything I know about this kind of task" |
| No pattern surfacing | Failure patterns (DEVELOPER fails 60% on git tasks) computed in agent-reputation.js but never injected into ARCHITECT context | ARCHITECT planned git tasks without knowing they had elevated failure risk |
| Success patterns ignored | `analyzeSuccesses()` computed but result discarded | No reinforcement of what worked |
| Dead on restart | Episode keyword index was in-process only; lessons.md read from disk each time | No semantic continuity across Render restarts |

## Solution Architecture

### memory-indexer.js

**What it does:**  
Maintains a persistent embedding index at `vault/12 Memory/memory-index.json`. Every episode stored via `_episodic.storeEpisode()` and every lesson generated by REFLECTOR is immediately added to the in-memory Map, then background-embedded (non-blocking via setImmediate).

**Key design decisions:**

1. **Local JSON, not Supabase.** `vault_embeddings` (Supabase) is owned by `langchain-rag.js`. Creating a separate table was unnecessary — the local vault path is faster (no network), simpler, and sufficient for 600 entries.

2. **Embedding text composition.** Episodes are embedded as: `"objective complexity:X failed_at:Y keyword1 keyword2..."` — not just the raw objective. Adding structural signals to the embedding text improves cosine similarity for same-failure-stage and same-complexity queries.

3. **Failure-first pruning.** When the episode cap (500) is exceeded, successful old episodes are pruned first. Failed episodes are retained longer — they encode system limitations that are high-value regardless of age.

4. **Deduplication.** `indexLesson()` hashes the first 100 chars. Identical lessons from repeated failures (e.g., "DEVELOPER routing returns empty when...") are silently skipped. Solves the noise accumulation problem identified in `long-term-memory-strategy.md`.

5. **Startup rebuild.** On first deploy (or Render restart where vault is fresh), `rebuildIndex()` triggers automatically if the index is empty. Scans `Episodes/` dir + `Lessons.md`. Keyword fallback serves retrieval while embeddings are computed in background.

### memory-retriever.js

**What it does:**  
Unified retrieval across all memory types. Single `retrieve(query, opts)` call returns `{ episodes, lessons, patterns, crossProject }`.

**Ranking formula:**

```
composite = similarity × 0.50
          + recency    × 0.25
          + successRate × 0.25

Where:
  similarity  = cosine(embed(query), embed(entry)) ∈ [0,1]
              OR keyword overlap ∈ [0,1] when embedding=null
  recency     = max(0.3, 1.0 - min(ageDays/90, 1) × 0.7)
  successRate = 1.0 for success, 0.0 for failure, 0.5 for lessons
```

**Why these weights:**
- Similarity at 50%: the primary signal — we want semantically relevant, not just recent
- Recency at 25%: needed because newer episodes reflect the current codebase state better
- SuccessRate at 25%: ARCHITECT should see successful examples first, but failed examples have diagnostic value (not filtered out entirely — just scored lower)

**Why not filter out failures?**  
Filtering to `successOnly` is available but off by default. A failure episode for a similar task is extremely useful context: "here's what went wrong last time, don't repeat it." The composite score naturally ranks successes higher while keeping failures visible.

## Retrieval Modes and Use Cases

### findSimilarEpisodes()
**Use case:** ARCHITECT context — what happened the last time we tried this kind of task?  
**Input:** task objective string  
**Output:** [{objective, success, complexity, failedStage, cost, _relevance, _method}]  
**Method indicator:** `_method: 'semantic'` | `'keyword'`

**Example query:** `"Add rate limiting to /api/voice-chat"`  
**Example semantic matches (vs keyword):**
```
Keyword: "Add rate limiting to /api/intelligence"  (exact word "rate", "limiting" match)
Semantic: "Protect voice endpoint from concurrent abuse" (conceptually same, no word overlap)
```

### findSimilarLessons()
**Use case:** Replace `getRecentLessons(12)` with task-relevant lesson retrieval  
**Input:** task objective string  
**Output:** [{text, type, _relevance, _method}]  
**Types returned:** `lesson` (from REFLECTOR) + `pattern` (indexed execution patterns)

**Example:** task = "Fix calendar sync timeout"  
Before: returns last 12 lessons regardless of topic  
After: returns lessons about timeout handling, external API calls, AbortController — regardless of position in Lessons.md

### findExecutionPatterns()
**Use case:** Surface known failure risk zones before execution  
**Input:** task objective string  
**Output:** [{stage, count, failureRate, examples, lastSeen}]  
**Derived from:** episode index (no separate storage, computed on demand)  
**Min threshold:** 2 occurrences before a pattern is returned (avoids noise from one-off failures)

**Example output:**
```
⚠ COMMITTER: 4 failures (40%) — git push rejected when worktree changes don't propagate
⚠ DEVELOPER: 3 failures (30%) — empty filesModified when target file > 20KB
```

This was previously available via `reflection-engine.js::analyzeFailures()` but was never injected into ARCHITECT context. It now surfaces automatically when `patterns: true` is passed to `retrieve()`.

### findCrossProject()
**Use case:** "Have we solved a similar problem in any other domain of the system?"  
**Input:** task objective string  
**Output:** { vault: { context, sources, confidence }, episodes: [...] }  
**Vault path:** delegates to `langchain-rag.js` which searches ALL vault domains (health, finance, uni, business, system, agents)  
**Episode path:** `findSimilarEpisodes()` across all episodes (no domain filter)

### retrieve() — Unified
**Use case:** Default ARCHITECT context injection  
**Parallel execution:** all enabled paths run via `Promise.allSettled` — no sequential blocking  
**Default config:** `{ episodes: true, lessons: true, patterns: false, crossProject: false }`

Enable patterns for tasks where `complexity === 'complex'` or `complexity === 'critical'`:
```js
const memCtx = await _retriever.retrieve(spec.objective, {
    episodes:  true,
    lessons:   true,
    patterns:  spec.complexity === 'complex' || spec.complexity === 'critical',
    episodeLimit: 3,
    lessonLimit:  5,
    patternLimit: 3,
});
```

## Embedding Infrastructure

**Primary:** Voyage AI `voyage-3-lite` (1024-dim) via `VOYAGE_API_KEY`  
**Fallback:** Gemini `gemini-embedding-001` (768-dim) via `GOOGLE_API_KEY`  
**Both:** Already configured and live in `lib/embed.js` — no changes needed

**Dimension note:** The index stores whatever dimension the embed API returns. Cosine similarity in `memory-retriever.js` checks `a.length === b.length` before computing — if a Voyage embed (1024-dim) is stored and a Gemini embed (768-dim) is queried (e.g. after API key rotation), the similarity returns 0 and keyword fallback activates automatically. No crash, no data corruption.

**Cost:**
- Voyage voyage-3-lite: ~$0.000002 per episode/lesson
- Gemini gemini-embedding-001: free tier (Google API key)
- 500 episodes + 100 lessons = 600 embed calls
- At Voyage pricing: $0.0012 for full index build — negligible

## Context Format — Before vs After

**Before (v2 keyword retrieval, 3 episodes max, 400 chars):**
```
SIMILAR PAST EXPERIENCES:
✓ Add OAuth login flow with JWT tokens [complex]
✗ Add OAuth token refresh endpoint [failed: DEVELOPER] [complex]
✓ Add JWT cookie auth for dashboard [moderate]
```

**After (v3 semantic retrieval, 3 episodes + 5 lessons, 500 chars):**
```
SIMILAR PAST EXPERIENCES:
✓ Add OAuth login flow with JWT tokens [complex] (0.847)
✓ Protect API endpoints from unauthorized access [moderate] (0.731)
✗ Implement session token rotation [failed:DEVELOPER] [complex] (0.694)

RELEVANT LESSONS:
• DEVELOPER must use oauth2:TOKEN format in git remote URL — fake username causes silent auth failure
• Auth routes must validate JWT sub field — missing sub breaks conversation identity resolution
• Session tokens must use base64url encoding not base64 — JWT decode fails silently otherwise
• Supabase RLS must be enabled via pg_database.js not runSQL() — SUPABASE_ACCESS_TOKEN not on Render
• REVIEWER catches missing try/catch on Supabase calls — always wrap .from().select()
```

The second version includes semantically matched lessons that are zero-keyword-overlap with the query ("auth routes", "base64url", "RLS") — which the v2 keyword system would have ranked last or not at all.

## Failure Pattern Context (when enabled)

For complex/critical tasks, patterns surface known risk zones:
```
FAILURE PATTERNS:
⚠ COMMITTER: 5 failures (45%) — git push rejected: remote contains work not in local
⚠ DEVELOPER: 3 failures (27%) — empty filesModified on large file routing
```

ARCHITECT can now plan defensively: allocate extra validation steps, use Sonnet model, add file-size checks before routing.

## Performance Characteristics

| Operation | Latency | Notes |
|-----------|---------|-------|
| `findSimilarEpisodes()` (semantic) | 50-500ms | embed API round-trip |
| `findSimilarEpisodes()` (keyword) | <1ms | pure in-process |
| `findSimilarLessons()` (semantic) | 50-500ms | same embed call reused |
| `findExecutionPatterns()` | <1ms | pure in-process aggregation |
| `findCrossProject()` | 100-800ms | vault RAG + episode search in parallel |
| `retrieve()` (all paths parallel) | ≈ max(individual) not sum | Promise.allSettled |
| `indexEpisode()` | <1ms | Map update + scheduleSave (disk write debounced 2s) |
| `indexLesson()` | <1ms | same |

**Critical path impact:** `retrieve()` is called once per pipeline run, inside the ARCHITECT context build which is already async. Total added latency on semantic path: 50-500ms, which is negligible vs the ARCHITECT Haiku/Sonnet call (2-10s). On keyword fallback: <1ms.

## Known Limitations

1. **Cold start embedding latency.** On fresh Render deploy, all 200+ existing episodes need embedding. During this ~5-7 min window, keyword fallback serves retrieval. This is expected and handled.

2. **Dimension mismatch on API rotation.** If `VOYAGE_API_KEY` is added after some entries were embedded with Gemini (768-dim), new entries will be 1024-dim. The dimension check in `_cosineSim()` will return 0 for mismatched pairs and fall back to keyword. To fix: call `rebuildIndex()` after switching embedding APIs to re-embed all entries with the new model.

3. **Index file not in git.** `memory-index.json` is stored in the vault (local filesystem), not in the git repo. On fresh clone/deploy, the index rebuilds from `Episodes/` dir. This is intentional — embeddings are computed artifacts, not source code.

4. **No embedding caching on query side.** The query `embedText(objective)` is called fresh each pipeline run. For identical repeated queries, this is a small avoidable cost (~$0.000002). Acceptable at current scale; add query-side LRU cache if hitting rate limits.
